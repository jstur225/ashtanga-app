## 任务目标
将“云端同步”假门测试的单一投票按钮拆分为四个细分选项，以精确收集用户对“数据同步”与“照片上传”这两个具体功能的需求权重。

## 详细步骤

### 第一步：修改 `handleVote` 逻辑 (FakeDoorModal.tsx)
- 修改 `handleVote` 接收 `choice: 'sync' | 'photo' | 'both'` 参数。
- 更新埋点：`trackEvent('vote_for_cloud_sync', { vote: 'yes', choice: choice })`。
- 保留 `setVotedCloud(true)` 和 `toast.success` 提示。

### 第二步：修改 `handleSecondary` 逻辑 (FakeDoorModal.tsx)
- 更新埋点：`trackEvent('vote_for_cloud_sync', { vote: 'no', choice: 'none' })`。

### 第三步：重构按钮 UI 区域 (FakeDoorModal.tsx)
- 在渲染部分，针对 `type === 'cloud'` 模式：
    1.  **移除** 原有的 `primaryBtn` 和 `secondaryBtn` 双按钮。
    2.  **新增** 垂直排列的四个按钮：
        - 按钮 A: 文案“【想要】云端数据同步”，点击触发 `handleVote('sync')`。
        - 按钮 B: 文案“【想要】体式照片上传”，点击触发 `handleVote('photo')`。
        - 按钮 C: 文案“【全都要】同步 + 照片”，点击触发 `handleVote('both')`。
        - 按钮 D: 文案“暂时不需要”，点击触发 `handleSecondary()`。
- **状态控制**: 
    - 当 `isVoted` 为 true 时，前三个按钮文案统一变为“已收到投票”，背景变绿并禁用。
    - 按钮 D 文案变为“收到啦”。

### 第四步：数据口径定义 (Analytics)
- **新事件格式**: `{ event: 'vote_for_cloud_sync', properties: { vote: 'yes'|'no', choice: 'sync'|'photo'|'both'|'none' } }`。
- **旧数据区分**: 历史数据中不含 `choice` 属性，在 Mixpanel 分析时使用 `choice is set` 过滤新版数据。

## 注意事项
- **禁止** 修改 `type === 'pro'` 的逻辑。
- **禁止** 修复 `StatsTab` 中 Pro Badge 的点击事件，保持其现有的不可点击状态。
